第二步：上线自动化交易（模拟盘运行）

这个阶段的核心任务不是赚钱，而是测试系统的“工程稳定性”和“异常处理能力”。

💣 核心深坑预警：

“假滑点”与“绝对成交”的幻觉（最致命）：

坑点： 模拟盘通常默认市场有无限流动性，只要市价触及你的委托价，立刻 100% 成交。但在实盘中，当你发出 1000 股的买单时，可能那个价位只有 300 股的卖单，导致你“部分成交”或“滑点”（以更高/更低的价格成交）。

对策： 在代码中强制加入“滑点测试”（例如买入价默认比触发价高一档，卖出价低一档），并编写应对“部分成交（Partial Fill）”和“撤单重报”的逻辑。

异步状态同步灾难（昨天刚经历过）：

坑点： 就像我们昨天修复的 Bug，脚本发出了卖出指令，但没有校验“可用余额”。在模拟盘中，你还会遇到网络延迟：指令发出去了，券商还没返回确认，脚本以为没买上，又发了一次，导致“重复下单”。

对策： 引入严格的“订单状态机（Order State Machine）”。只有在收到 API 返回的 ORDER_FILLED（已成交）状态后，才能更新本地仓位继续下一步操作。

API 频率限制与断线重连：

坑点： 券商的 API 通常有频率限制（比如每秒最多查询 3 次）。如果你的脚本在行情剧烈时死循环查询，会被券商直接封停 IP。

对策： 增加请求休眠（time.sleep）机制，并必须写好 try...except 应对断网、接口超时等意外，确保断线重连后脚本能自动恢复状态。

⏱️ 预估周期：2 ~ 4 周

前 1-2 周： 主要在修 Bug（处理各种报错、未捕获异常、废单）。

后 1-2 周： 脚本能在无人值守的情况下，稳定运行两周，且本地记录的仓位与模拟盘实际仓位做到 100% 账实相符。

第三步：计算收益率与持续优化

这是量化交易的终身课题，是从“程序员”进化为“宽客（Quant）”的阶段。

💣 核心深坑预警：

被摩擦成本（手续费与印花税）吸血：

坑点： 做 T 的特点是高频、微利。在模拟盘里你觉得每天做 T 能赚 2%，但扣掉 A股的印花税（千分之0.5，仅卖出收取）、券商佣金（万分之二点五，有 5 元最低消费），你的实际利润可能会被直接腰斩甚至变成负数。

对策： 在收益率计算模块中，必须硬编码扣除所有税费。如果单笔利润覆盖不了 5 元的手续费最低消费，脚本必须放弃这次做 T。

过度拟合（Overfitting）的陷阱：

坑点： 你发现如果把 RSI 参数从 20 改成 18，过去一个月的收益率提高了 50%。于是你上线实盘，结果亏得一塌糊涂。因为你是在拿历史数据“对答案”，市场环境一旦改变（比如从震荡市变成单边熊市），旧参数就会失效。

对策： 永远保留一部分“盲测数据”（Out-of-sample test）。不要追求历史回测的完美曲线，追求逻辑的普适性。

极端行情的黑天鹅：

坑点： 股票突然涨停/跌停，买不进也卖不出；或者像今天早盘一样，个股开盘直接跳空低开 5%。你的做 T 脚本如果还是机械地逢低买入，就会把子弹打光，变成被动重仓满仓。

对策： 加入“大盘环境熔断”和“单日亏损熔断”机制。比如：当大盘跌幅超过 2%，或者单只股票日内跌幅超过 7%，立刻停止一切买入动作（只卖不买）。

⏱️ 预估周期：3 ~ 6 个月（至第一代成熟版本）

第 1 个月： 经历一轮完整的震荡市、单边上涨和单边下跌行情，收集实盘/模拟盘的盈亏数据。

第 2-3 个月： 根据数据，剔除掉那些“胜率高但盈亏比极差”的无效交易信号，优化买卖点的颗粒度。

后续： 这是一个持续一生的过程。市场在进化，你的策略库也需要不断扩充（比如从单一的 RSI 做 T，升级为结合量价背离、均线共振的多因子模型）。

教练的下一步建议：
在推进第二步（模拟盘）时，第一要务是建立完善的日志系统（Logging）。脚本的每一次查询、下单、撤单、报错，都必须详细记录时间戳和状态码。当出现幽灵 Bug 时，日志是你唯一的救命稻草。
